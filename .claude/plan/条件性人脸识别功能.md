# 功能规划文档：条件性人脸识别系统 (Conditional Face Detection)

## 1. 功能概述

### 1.1 背景与目标
为了提升用户生成图片的质量并减少无效任务对 GPU 资源的消耗，系统需要在用户上传照片阶段进行人脸识别验证。由于不同业务领域（Domain）对人脸的要求不同（如：婚纱照必须有人脸，而风景画不需要），我们需要根据 `/api/domains` 返回的配置，动态开启或关闭人脸检测逻辑。

### 1.2 核心需求
- **配置驱动**：根据领域模型中的 `require_face_detection` 字段决定验证流程。
- **异步验证**：在照片上传的同时进行人脸识别，互不阻塞。
- **状态反馈**：在 UI 上清晰展示"识别中"、"验证成功"或"未检测到人物"的状态。
- **流程阻断**：若领域要求人脸识别且识别失败，阻止用户进入下一步生成阶段。

---

## 2. 技术方案

### 2.1 架构设计
采用**配置下发 -> 逻辑分流 -> 状态反馈**的闭环架构。

1.  **数据层**：`useDomains` 钩子负责从后端同步最新的 `require_face_detection` 配置。
2.  **逻辑层**：`useParallelPhotoUpload` 钩子封装核心逻辑，根据配置参数决定是否调用识别接口。
3.  **表现层**：`StepUpload` 组件根据 `identifyStatus` 渲染差异化 UI。

### 2.2 数据流设计
```
StepFlow (获取 domains)
  ↓
StepUpload (传递 requireFaceDetection)
  ↓
useParallelPhotoUpload (逻辑分流)
  ↓
  ├─ require_face_detection = true  → 调用 /api/identify-image
  └─ require_face_detection = false → 直接标记为 valid
  ↓
更新 PhotoState.identifyStatus
  ↓
UI 渲染差异化状态
```

```typescript
async function identifyPerson(
  dataUrl: string
): Promise<{ hasPerson: boolean; description: string }> {
  const res = await fetch('/api/identify-image', {
    method: 'POST',
    credentials: 'include',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ images: [dataUrl] }),
  });

  if (!res.ok) {
    throw new Error('识别服务失败');
  }

  const data = await res.json();
  const result = data.results?.[0];

  return {
    hasPerson: result?.hasPerson ?? false,
    description: result?.description ?? '识别失败',
  };
}
```

#### 修改点 3：逻辑分流
```typescript
const identifyPromise = (async () => {
  // 如果不需要人脸识别，直接标记为 valid
  if (!requireFaceDetection) {
    dispatch({
      type: 'UPDATE_PHOTO',
      photoId: photo.id,
      updates: { identifyStatus: 'valid' },
    });
    return;
  }

  // 需要人脸识别，执行真实 API 调用
  dispatch({
    type: 'UPDATE_PHOTO',
    photoId: photo.id,
    updates: { identifyStatus: 'identifying' },
  });

  try {
    const result = await identifyPerson(photo.dataUrl);
    dispatch({
      type: 'UPDATE_PHOTO',
      photoId: photo.id,
      updates: {
        identifyStatus: result.hasPerson ? 'valid' : 'invalid',
        identifyDescription: result.description,
      },
    });
  } catch (err) {
    dispatch({
      type: 'UPDATE_PHOTO',
      photoId: photo.id,
      updates: {
        identifyStatus: 'error',
        identifyDescription: err instanceof Error ? err.message : '识别失败',
      },
    });
  }
})();
```

## 5. 风险评估

| 风险点 | 影响程度 | 应对措施 |
|--------|----------|----------|
| **识别接口高延迟** | 中 | 在 UI 上展示明显的加载动画，增加超时处理机制（15s） |
| **接口误判 (False Negative)** | 高 | 在识别失败后，允许用户查看原因，并提供"重试"入口 |
| **API 算力瓶颈** | 中 | 后端增加 Rate Limiting，前端在 429 时提示用户稍后再试 |
| **类型安全问题** | 低 | 严格遵循 TypeScript strict mode，禁止 any 类型 |
