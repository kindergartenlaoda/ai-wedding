# 系统提示词长什么样？OpenClaw 如何组装一份"可控的 AI 运行说明书"

> 你以为和 AI 聊天就是"你问我答"？其实在你发出每条消息之前，AI 已经读完了一本厚厚的"运行说明书"——这本说明书决定了它是谁、能做什么、怎么回答你。今天我们通过开源项目 OpenClaw，完整拆解这份"说明书"的真实面貌和组装过程。

---

## 一、你可能从未见过的"隐藏文本"

当你在 ChatGPT 或 Claude 的对话框里输入"你好"时，模型实际收到的不只是"你好"这两个字。在它之前，还有一大段你看不到的文本——**系统提示词（System Prompt）**。

这段文本告诉模型：

- 你叫什么名字，该用什么语气说话
- 你能调用哪些工具，不能做哪些事
- 现在是什么时间，在什么环境下运行
- 用户是谁，有什么偏好需要注意
- 之前聊过什么，哪些记忆需要带上

**系统提示词就是 AI 的"运行说明书"。** 它不是一句话，而是一个精心设计的多层结构。

OpenClaw 是一个开源的 AI 智能体网关项目，它把这个组装过程完全透明化了。接下来，我们就跟着它的源码逻辑，看看一份完整的系统提示词到底长什么样。

---

## 二、一份真实的系统提示词，长这样

先看全貌。OpenClaw 每次运行时组装的系统提示词，大约 **38,000 字符（约 9,600 Token）**，由以下 13 个模块拼接而成：

| 序号 | 模块 | 说明 |
|:---:|------|------|
| 1 | **Reasoning** | 推理模式与可见性级别 |
| 2 | **Runtime** | 运行环境（OS/模型/目录） |
| 3 | **Heartbeats** | 心跳机制与确认行为 |
| 4 | **Reply Tags** | 回复标签语法（按提供商） |
| 5 | **Date & Time** | 时区信息（不含动态时钟） |
| 6 | **Sandbox** | 沙箱运行指示（可选） |
| 7 | **Workspace** | 工作目录位置 |
| 8 | **Documentation** | 文档路径 |
| 9 | **Self-Update** | 自更新指令 |
| 10 | **Skills** | 技能列表（仅元数据） |
| 11 | **Safety** | 安全护栏提醒 |
| 12 | **Tooling** | 工具列表 + 描述 |
| 13 | **Project Context** | 注入的工作区文件 |

> 📊 合计约 **38,412 字符 / ~9,603 Token**，其中 Project Context 占 **~5,976 Token**

你可以用 `/context list` 命令看到这个实时分解：

```
🧠 Context breakdown
System prompt (run): 38,412 chars (~9,603 tok)
  (Project Context 23,901 chars (~5,976 tok))

Injected workspace files:
- AGENTS.md:    OK        | raw 1,742 chars | injected 1,742 chars
- SOUL.md:      OK        | raw   912 chars | injected   912 chars
- TOOLS.md:     TRUNCATED | raw 54,210 chars | injected 20,962 chars
- IDENTITY.md:  OK        | raw   211 chars | injected   211 chars
- USER.md:      OK        | raw   388 chars | injected   388 chars
- HEARTBEAT.md: MISSING   | raw     0       | injected     0

Skills list: 2,184 chars (~546 tok) (12 skills)
Tool schemas (JSON): 31,988 chars (~7,997 tok)
```

注意最后一行——**工具 Schema 占了将近 8,000 Token**，这是你在对话中完全看不到、但实实在在消耗预算的"隐形成本"。

## 三、逐层拆解：每个模块在干什么

### 第一层：运行环境感知

```
Runtime: macOS 15.2 | Node 22.x | Model: claude-opus-4-6
Repo root: /Users/me/project
Thinking: medium
```

这一层告诉模型"你在哪"。包括操作系统、Node 版本、当前使用的模型、仓库根目录、思考级别。只有一行，但至关重要——模型需要知道自己跑在 macOS 还是 Linux 上，才能给出正确的 shell 命令。

时间信息单独处理：**只注入时区，不注入动态时钟**。为什么？因为系统提示词会被缓存，如果每秒都变，缓存就失效了。当模型真的需要当前时间时，通过 `session_status` 工具获取。

### 第二层：人格与身份（Project Context）

这是系统提示词中**占比最大**的部分，由一组工作区 Markdown 文件注入：

| 文件 | 干什么 | 生活类比 |
|------|--------|---------|
| `AGENTS.md` | 操作指令、行为规则、优先级 | 员工手册 |
| `SOUL.md` | 人设、语调、边界 | 性格说明书 |
| `USER.md` | 用户是谁、怎么称呼 | 客户档案 |
| `IDENTITY.md` | 智能体的名称和风格 | 名片 |
| `TOOLS.md` | 本地工具使用约定 | 工具箱说明 |
| `HEARTBEAT.md` | 心跳运行清单 | 每日待办 |
| `MEMORY.md` | 精选的长期记忆 | 备忘录 |

**这是 OpenClaw 最核心的设计理念：用 Markdown 文件定义 AI 的一切行为。**

想让 AI 说话更正式？改 `SOUL.md`。想让它记住你喜欢用 pnpm 而不是 npm？写进 `MEMORY.md`。想限制它不能执行危险命令？调整 `AGENTS.md`。

不需要写代码，不需要调 API，编辑几个文本文件就能重塑一个 AI 助手的"人格"。

### 第三层：技能目录（按需加载）

系统提示词中包含一个紧凑的技能列表，但**只有元数据，没有具体指令**：

```xml
<available_skills>
  <skill>
    <name>frontend-design</name>
    <description>Create production-grade frontend interfaces</description>
    <location>~/.openclaw/skills/frontend-design/SKILL.md</location>
  </skill>
  <!-- ... 11 more skills -->
</available_skills>
```

12 个技能的列表只占 ~546 Token。当模型判断需要某个技能时，它会主动 `read` 对应的 `SKILL.md` 文件加载完整指令。

**这个设计非常聪明**：如果把所有技能的完整指令都塞进系统提示词，可能要多花几千甚至上万 Token；按需加载则把基础成本压到最低。

### 第四层：工具声明

工具以两种形式影响上下文：

1. **系统提示词中的文本描述**（~258 Token）：一行一个工具名 + 简短说明
2. **JSON Schema**（~7,997 Token）：发送给模型的完整工具调用规范

第二种你完全看不到，但它是上下文中最大的单项开销之一。`/context detail` 可以揭示哪个工具的 Schema 最大：

```
Top tools (schema size):
- browser: 9,812 chars (~2,453 tok)
- exec:    6,240 chars (~1,560 tok)
```

仅 `browser` 一个工具就吃掉了 2,453 Token。

### 第五层：安全护栏

系统提示词中包含简短的安全提醒，引导模型避免越权行为。但 OpenClaw 的文档明确指出：

> 系统提示词中的安全护栏是**建议性的**。它们引导模型行为但不强制执行策略。

真正的硬性安全由系统级机制保证：工具策略（允许/拒绝）、exec 审批、沙箱隔离、通道白名单。**软引导 + 硬执行，两层分离。**

## 四、组装流程：一份提示词是怎么"拼"出来的

现在我们知道了系统提示词"长什么样"，接下来看它是**怎么组装**的。

当你发送一条消息时，OpenClaw 内部的组装流程如下：

```
消息到达网关
  ↓
① 获取会话写锁（防止并发冲突）
  ↓
② 解析工作区引导文件
   ├── 读取 AGENTS.md, SOUL.md, USER.md, IDENTITY.md...
   ├── 每个文件最多 20,000 字符，超出截断
   └── 所有文件合计最多 24,000 字符
  ↓
③ 加载技能快照
   └── 扫描可用技能，生成元数据列表
  ↓
④ 构建系统提示词
   ├── 基础模板（Runtime, Safety, Tooling...）
   ├── + 注入引导文件 → Project Context
   ├── + 注入技能列表 → Skills
   └── + 运行时覆盖（时区、沙箱状态等）
  ↓
⑤ 拼接最终上下文
   ├── 系统提示词（~9,600 tok）
   ├── + 对话历史（或压缩摘要）
   ├── + 工具 Schema（~8,000 tok）
   └── + 当前用户消息
  ↓
⑥ 发送给模型
  ↓
⑦ 模型推理 → 可能调用工具 → 多轮循环
  ↓
⑧ 流式输出回复
  ↓
⑨ 持久化会话 → 检查是否需要压缩
```

**每次运行都重新组装。** 系统提示词不是一次性写死的，而是每次都根据当前状态动态构建。你改了 `SOUL.md`，下一条消息立刻生效。

### 组装过程中的三个关键决策

**决策一：注入多少？**

引导文件有严格的 Token 预算：

- 单文件上限：20,000 字符（`bootstrapMaxChars`）
- 总量上限：24,000 字符（`bootstrapTotalMaxChars`）

超出的文件会被截断并标记 `TRUNCATED`。这意味着你的 `TOOLS.md` 如果写了 5 万字，实际只有前 2 万字会被模型看到。

**决策二：注入给谁？**

OpenClaw 支持三种提示词模式：

| 模式 | 场景 | 包含内容 |
|------|------|---------|
| `full` | 主智能体 | 全部 13 个模块 |
| `minimal` | 子智能体 | 省略 Skills、Memory、Self-Update 等 |
| `none` | 极简 | 仅基础身份行 |

子智能体不需要知道"我有哪些技能"或"怎么更新自己"，只需要知道"我能用什么工具"和"我在哪个目录"。**精简提示词 = 节省 Token + 减少干扰。**

**决策三：钩子拦截**

在系统提示词最终确定之前，有一个 `agent:bootstrap` 钩子可以拦截并修改注入的引导文件。比如：

- 把 `SOUL.md` 替换为另一个人设文件
- 根据时间段注入不同的行为规则
- 为特定通道（如群聊）过滤掉敏感的引导内容

这让系统提示词的组装变成了一个**可编程的管道**，而不是死板的模板填充。

## 五、对话太长怎么办：压缩与记忆的协作

系统提示词组装好了，但上下文窗口是有限的。随着对话越来越长，总会有装不下的时候。OpenClaw 用两个机制来应对：

### 压缩（Compaction）：给旧对话"做摘要"

当会话接近上下文窗口上限时，OpenClaw 自动触发压缩：

```
较旧的对话 → 摘要化为紧凑条目 → 保持最近消息完整
```

压缩后的摘要会**持久化**到会话的 JSONL 文件中。后续请求使用的是：压缩摘要 + 最近的完整消息。

### 记忆刷新：压缩前的"抢救"

更精妙的是，在压缩**之前**，OpenClaw 会触发一个静默的记忆刷新轮次：

```
会话接近上限
  ↓
触发静默轮次，提醒 AI：
"Session nearing compaction. Store durable memories now."
  ↓
AI 把重要信息写入 memory/YYYY-MM-DD.md
  ↓
AI 回复 NO_REPLY（用户看不到这个轮次）
  ↓
执行压缩
```

这样即使旧对话被压缩了，关键信息已经安全地写入了磁盘上的记忆文件。

### 记忆的两层设计

| 层级 | 文件 | 注入方式 | 适合存什么 |
|------|------|---------|-----------|
| 长期记忆 | `MEMORY.md` | 每次自动注入（消耗 Token） | 决策、偏好、持久事实 |
| 每日日志 | `memory/YYYY-MM-DD.md` | 不自动注入，按需搜索 | 日常笔记、运行上下文 |

`MEMORY.md` 每次都会被塞进系统提示词，所以要**保持精简**——它越长，留给对话的空间就越少，压缩也会更频繁。

每日日志则通过 `memory_search`（支持向量语义搜索）按需检索，不占基础 Token 预算。

## 六、上下文的完整账本

把所有东西加在一起，一次典型的 AI 运行，上下文窗口里装的是：

```
┌──────────────────────────────────────┐
│         模型上下文窗口（如 32K）       │
├──────────────────────────────────────┤
│  系统提示词           ~9,600 tok     │
│    ├── 基础模板       ~3,600 tok     │
│    └── Project Context ~6,000 tok    │
│  工具 Schema (JSON)   ~8,000 tok     │
│  对话历史/压缩摘要     ~12,000 tok    │
│  当前消息 + 附件       ~2,000 tok     │
│  提供商隐藏头          ~400 tok       │
├──────────────────────────────────────┤
│  合计                 ~32,000 tok    │
└──────────────────────────────────────┘
```

**系统提示词 + 工具 Schema 就占了约 17,600 Token，超过窗口的一半。** 这就是为什么 OpenClaw 如此重视精简：每省下 1,000 Token 的提示词空间，就多出 1,000 Token 给对话历史。

## 七、设计哲学：五个值得借鉴的原则

回到标题——**系统提示词长什么样？**

它不是一段话，而是一个**精心设计的多层架构**：

```
系统提示词
├── 运行时环境（我在哪台机器上？用什么模型？）
├── 人格档案（我是谁？怎么说话？）
├── 用户画像（对面是谁？怎么称呼？）
├── 操作指令（我该怎么做？优先级是什么？）
├── 工具清单（我能用什么工具？）
├── 技能列表（我有哪些专业知识可以调用？）
├── 记忆注入（我记得什么？）
├── 安全护栏（我不该做什么？）
└── 时间感知（现在几点？什么时区？）
```

**怎么组装？** 每次运行时动态构建，经过引导文件注入、Token 预算裁剪、提示词模式选择、钩子拦截四个阶段，最终拼成一份"刚好够用"的运行说明书。

从 OpenClaw 的设计中，我提炼出五个值得所有 AI 开发者借鉴的原则：

**1. 可控性——用 Markdown 定义一切**
不需要改代码，编辑文本文件就能重塑 AI 行为。降低了定制门槛，也让非技术人员能参与调优。

**2. 透明性——让用户看到 AI "看到了什么"**
`/context list` 和 `/context detail` 把黑盒打开。你能精确知道每个文件注入了多少 Token、哪个工具的 Schema 最大、总预算用了多少。

**3. 经济性——每个 Token 都精打细算**
技能按需加载而非全量注入；引导文件有严格的字符上限；时间信息只给时区不给动态时钟以保护缓存；子智能体用精简提示词。处处都在省 Token。

**4. 动态性——每次运行都重新组装**
系统提示词不是写死的模板，而是根据当前状态（时间、工作区文件、技能、沙箱配置）动态构建。改了文件，下一条消息立刻生效。

**5. 分层安全——软引导 + 硬执行**
系统提示词里的安全提醒是"建议"，真正的防线是工具策略、沙箱隔离、exec 审批这些系统级机制。不把安全全押在提示词上。

## 写在最后

好的 AI 产品，不是把最强的模型接上去就完了。**模型的能力上限由参数决定，但模型的行为下限由系统提示词决定。**

OpenClaw 用一个开源实现告诉我们：系统提示词不是一句"你是一个有帮助的助手"，而是一份经过精心设计的、有预算约束的、可编程的、动态组装的运行说明书。

下次当你和 AI 对话时，不妨想想——在你的消息之前，它已经读了将近一万个 Token 的"说明书"。而这份说明书的质量，决定了它能不能真正帮到你。
